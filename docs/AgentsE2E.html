Agents intro User stories Sections

Brief summary

  • AI agents leverage LLMs either in your own AI Landing Zones or as part of a managed solution.
  • They need Runtime to operate and must be hosted somewhere.
  • Agents need to access various tools to take actions and knowledge for grounding.
  • Users interact with agents from different experiences using channels.
  • Agents typically need to work with identity to support user-delegated permissions, but they also have their own identity for autonomous tasks (to access tools, knowledge, or other agents).
  • All of this should be governed by a control plane to provide administration, observability, catalog, onboarding, security, and more.
  • Security must understand the current agent landscape, posture, conditional access policies for agents, responsible AI, and data security.
  • Successful development and continuous improvement of agents require strong evaluations and lifecycle management with CI/CD.

Reference architecture sections

Models can be accessed via APIs from different runtimes and hosting options including Foundry Agents and Copilot Studio, but can also be provided as part of the platform.

A large-scale enterprise AI landing zone uses Azure API Management as an AI gateway to manage access to LLMs. This includes clever routing and capacity policies, access control, auditing capabilities, inline security enforcement, and a data plane proxy for MCP tools.

Foundry provides resources and projects, plus UI experiences such as playgrounds and an agent builder. Support for bring-your-own APIM with full UI support is planned.

Foundry also provides a developer-oriented experience with observability, evals, a model catalog, an agents catalog, and a tools catalog for discovery.

Responses API

Can be considered a lightweight runtime, but this story focuses on more advanced ones.

Copilot Studio

A no-code platform that leverages Foundry runtime under the covers.

Foundry Agent Service

A developer-oriented platform ranging from declarative agents and workflows all the way to hosting other runtimes.

Microsoft Agent Framework

An open-source framework, very strong in Python and C#, sponsored by Microsoft to build agents, multi-agent systems, and orchestration workflows that can be hosted anywhere.

LangGraph

Another open-source framework, strong in Python, modern and low-level, with some higher-level constructs from LangChain. It has a community-driven clone for Java.

Hosted options

Copilot Studio hosts only Copilot Studio agents, but it can interact with other agents and tools outside of it.

Foundry Agent Service supports multiple runtimes, such as declarative Foundry agents, hosting of Microsoft Agent Framework or LangGraph, and others via SDK integration and a Docker image.

Self-hosted options

Can run anywhere, for example Azure Container Apps or Azure Kubernetes Service, and can use the Microsoft 365 Agents SDK to integrate into governance, security, and Microsoft experiences.

Authentication and authorization for tools is critical and should be based on Entra ID identities and scopes. MCP should publish scopes in Entra ID, and users via OBO plus agents via Agent ID should request the proper scope.

Granular authorization is important: an MCP server can parse token claims such as roles or security groups and enforce different policies. MCP is often a facade over existing APIs, so authn/authz should be implemented at the API level; Azure API Management can enforce it where backends are not ready.

Data plane policies protect backend systems from overuse, implement routing and version control, enforce auditing and logging, and apply network restrictions. These can be handled by Azure API Management, including republishing an API as MCP or providing a data plane proxy.

A catalog of tools is available as part of Agent 365, which works as a tool gateway and is visible in Microsoft agent development tools such as Copilot Studio. Microsoft provides built-in MCP tools such as Outlook Email, Teams, SharePoint and OneDrive, Outlook Calendar, Microsoft Dataverse, and others.

Azure API Center is another extensible catalog option. It can import MCPs from Azure API Management, supports a web-based catalog with RBAC, and can be accessed programmatically by Foundry Agent Service or GitHub Copilot private MCP lists.

All IQs provide a simple MCP-based interface for agents to access complex knowledge without understanding the complexity of underlying data sources and retrieval techniques.

WorkIQ

Brings knowledge about people and work: emails, calendar, chats, meetings, and more.

Foundry IQ

Brings institutional knowledge about your organization: documents, wikis, databases, policies, and more.

You can build processing pipelines that ingest knowledge from various sources, including poorly structured data such as PDFs, and build indexes in AI Search. Agents use Foundry IQ, which implements agentic search such as query rewriting, query decomposition, semantic search, keyword search, BM25, semantic ranking, and filtering.

Fabric IQ

Brings semantic knowledge about your business: customers, products, inventory, transactions, and more. Data teams manage data and semantic layers so agents can understand business data and use data agents for questions about business context.

Channels require more than messages: reasoning events, typing, commands, multi-modality, and declarative UI extensions such as Adaptive Cards.

They should require strong authentication and authorization tunnels, provide user on-behalf-of flow, and sometimes expose internal agents securely to public internet channels such as Copilot or Teams through a proxy or relay.

In Microsoft's story, this is handled with Activity Protocol and the Microsoft 365 Agents SDK, and automatically provided by Copilot Studio and Foundry Agent Service for supported agent types.

Agents published to Microsoft 365 have identities assigned automatically and permissions managed in Agent 365. Underlying Microsoft Entra capabilities can be used for other agents too, although they require manual permission and access policy configuration.

Agents support an on-behalf-of user authentication flow, where Azure Bot Service with the Microsoft 365 Agents SDK or Agent 365 SDK handles token exchange and OBO flow management. The SDK delivers access tokens for tools and services such as custom MCP servers, Microsoft Graph, or Microsoft MCP tools.

Agent identity, similar to app registration client credentials but curated for agents, is used for background agents without user interaction and for agent-to-agent scenarios. In the future, an agent user ID may let agents act as digital employees with their own Outlook, Teams, notifications, and more.

Agents can be published in Microsoft 365 with icon, description, required permissions, and metadata. Users discover them in Microsoft 365 and use them across Microsoft 365 Copilot, Teams, and other channels. Administrators can manage access and monitor usage.

All agents built in Copilot Studio and Foundry Agent Service can be published to the Microsoft 365 ecosystem. Agents built with any framework and self-hosted can be published when instrumented with the Microsoft 365 Agents SDK.

The Agent 365 SDK allows any agent to integrate into identity, auditing, observability, security, and tool access in Agent 365, plus forward-looking digital worker capabilities such as an agent user ID with email, Teams, Word comments, notifications, meetings, and mentions.

Agent 365 uses blueprints: agent templates with predefined capabilities, permissions, and tool access. Users can request instantiation of an agent for their use, which creates a specific instance with its own Agent ID in Entra ID and consented or fine-tuned permissions.

Coding agents are different but important. Expect less off-the-shelf software and more custom code written by agents. GitHub Copilot CLI, Claude Code, CODEX, cloud-based GitHub agents, GitHub Agents HQ, and GitHub Agentic Workflows are part of the control-plane story for software delivery agents.

Developers need to understand performance, issues, bottlenecks, and improvement opportunities. Observability includes traces of conversations, tool calls, multi-agent turns, time, latency, user ID, role, group, tenant, and more.

Industry is standardizing on OpenTelemetry for agents with GenAI semantic conventions.

Copilot Studio and Foundry Agent Service provide built-in logging, metrics, tracing, and access to Application Insights so customers can query raw data and build dashboards in Azure Monitor, Grafana, or Power BI.

Custom OpenTelemetry endpoints are available for hosted agents in Foundry such as Microsoft Agent Framework or LangGraph, and for self-hosted agents connected via Microsoft 365 Agents SDK. You can use an OpenTelemetry Collector to distribute telemetry to systems such as Foundry, LangFuse, or Grafana Tempo.

Agent 365 acts as an OpenTelemetry collector and provides observability to published agents or custom agents instrumented with the Agent 365 SDK. Data is used by Agent 365, Defender for AI, Purview, and Microsoft Sentinel. As of February 2026, customers cannot yet access raw telemetry directly.

Strategy: instrument once, observe everywhere. Vision for 2026: intelligent observability where agents look after other agents.

AI agents inherit Microsoft Purview protections for data classification, sensitivity labels, DLP, and auditing. Agents respect user and agent identity-based access controls and are subject to the same encryption and compliance policies as users.

Purview integrates with Microsoft 365, Azure, and Agent 365 to enforce zero trust, prevent oversharing, and ensure auditability. Agent-specific risk indicators and observability are available via Agent 365 and Purview dashboards.

Microsoft AI Guardrails provide inline protection against harmful content, PII oversharing, or jailbreaking risks. Because they sit in the flow of agent-to-LLM interactions, they can block or annotate risks in real time.

Microsoft Defender for AI provides post-hoc detection capabilities and can detect more complex attack patterns. It also brings posture management capabilities with risk indicators and recommendations for mitigation. Defender for AI reports to Defender for Cloud and Microsoft Sentinel for SOC correlation.

Copilot Studio

Has built-in eval capabilities.

Foundry Agent Service

Comes with rich built-in evaluations and red teaming capabilities, including offline and batch evals, continuous evaluations, and CI/CD integration.

All Foundry agent types support API access and can therefore be evaluated with eval frameworks such as DeepEval or LangFuse.

Copilot Studio supports CI/CD capabilities via Power Platform ALM.

Foundry Agent Service fully supports any CI/CD for all types of agents, and even declarative agents can be managed effectively via SDK.
