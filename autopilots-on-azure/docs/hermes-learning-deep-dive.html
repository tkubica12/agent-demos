01 / THE CONTRACT

First teach the distinction.

The Role Blueprint's SOUL.md is the sole authority for the Worker's stable identity and adaptation classification. The bridge points every request mode to that authority and adds only transport-specific safety and provenance protocol.

ROLE BLUEPRINT · SOUL.MD

Durable role instruction

Small private facts go to Personal Memory. Rich named context becomes a Private Playbook. Corrections to inherited behavior patch a Role Skill. New reusable capabilities become Candidate Improvements.

BRIDGE · PER REQUEST

Policy reference + protocol

The bridge tells the model to follow the active SOUL.md rather than restating its taxonomy. It separately requires the machine-readable provenance block that reconciliation needs after the user-visible answer.

The model proposes the change; the runtime verifies and commits it.

The model creates or updates a learning artifact, such as a skill, and emits provenance explaining why. The runtime compares the files before and after the turn to determine exactly what changed. If the change and provenance agree and pass privacy checks, the runtime packages the authoritative details as one row in records.jsonl. Otherwise, it rolls the governed change back.

What exists, where it lives, and who provides it.

Hermes supplies memory injection, Work History, native skills, progressive disclosure, and learning tools. Our solution adds the Role Blueprint policy, namespace meanings, transactional governance, provenance, export, and release lifecycle.

How each kind of state reaches the model

Hermes profile + Role Blueprint

Fresh-session context

SOUL.md defines role behavior and adaptation policy. Hermes injects the compact USER.md and MEMORY.md snapshot into each fresh session.

Hermes engine + solution categories

Progressive skill context

Hermes indexes skill names and descriptions. Full SKILL.md instructions load only when relevant or explicitly invoked; our solution gives role, private, and candidate namespaces their lifecycle meaning.

Hermes native

Retrieved history

Work History is not injected wholesale. Hermes uses session recall/search and Dreaming to retrieve relevant prior work. Raw history remains private and is never copied into a Learning Packet.

Why the ownership distinction helps

It prevents us from describing native Hermes capabilities as custom code. This repository does not reimplement memory, Work History, skill discovery, or skill_manage. It packages the profile and adds governance around those mechanisms: classification, reserved namespaces, snapshots, rollback, provenance, privacy scanning, export, and refresh.

03 / LEARNING WINDOWS

Reflect across a wider horizon.

Dreaming is not a separate output namespace and it does not approve its own work. It is an explicit Hermes run guided by a reviewed Role Skill and constrained by the same classifications, provenance rules, and transaction boundary.

Foreground learning in one sentence

An ordinary turn may use native memory or skill tools; exact /learn <instruction> selects constrained learning instructions for one transactional model call. Dreaming is the later second chance across Work History, not a retry of that turn.

1

The bridge creates a focused Dream request

It sets a maximum number of governed artifacts and requires one provenance object per Role Skill or Candidate Improvement.

2

The Dreaming Role Skill drives reflection

Review sessions, outcomes, corrections, failed approaches, memories, playbooks, skills, candidates, and quarantine observations; then classify or intentionally store nothing.

3

The normal transaction reconciles output

The adapter invokes Dreaming as source dream, then reads runtime learning status and returns the accepted and rejected records.

4

Scheduling reuses the same path

The coordinator creates a unique Dream session, retries transient failures, and prepares a packet only when learning records exist.

04 / TRANSACTIONAL GOVERNANCE

Trust the diff, not the narration.

Provenance is not accepted merely because the model emitted JSON. The runtime derives changed artifacts from before/after snapshots and requires an exact one-to-one match.

SNAPSHOT + LEASE

Before the model

A Worker holds one learning lease. The runtime recovers pending work and snapshots all governed and private skill files.

MODEL PROPOSAL

During the turn

The model changes skills through native tools and returns a hidden provenance proposal after its visible answer. It never writes the journal directly.

RUNTIME RECORD

After the turn

The reconciler derives the real diff and action, validates privacy and provenance, then enriches accepted records with hashes, Worker identity, and Role Release.

ATOMIC DECISION

Before the reply

Accepted records are appended through a temporary file and atomic replace. Rejected governed changes are restored before the lease is released.

05 / LEARNING PACKET

Extract only eligible learning.

Packet extraction reconstructs changed governed artifacts from current files, binds each to provenance with the same final hash, privacy-scans the actual content, and excludes the entire private plane.

Role + candidate diffsComputed against the immutable Role Skill baseline.
Matching provenanceThe record's afterHash must equal the current artifact hash.
Private exclusionsMemory, sessions, private skills, workspace, credentials, logs, and quarantine stay out.
Pending packetExact content plus digest; approval is still required.
PREPARE

Content-addressed extraction

The pending packet includes files, provenance, Role Release, Worker assignment, a governed-state hash, privacy status, and a digest returned for review.

ATTEST + EXPORT

Approval binds exact state

The receipt must match the packet digest, Worker, release commit, and current governed-state hash, then pass Ed25519 signature verification. Later skill changes invalidate the approved packet.

A Learning Packet is not shared behavior.

It is an approved export candidate for Collective Learning Review. Shared behavior changes only after downstream review, Promotion, merge, and issuance of a newer immutable Role Release.

06 / WORKER REFRESH

Adopt shared learning without losing identity.

Refresh is forward-only. The new Role Release replaces distribution-owned behavior, while the Worker's private continuity remains on its profile and Data Disk.

PRESERVED

Worker-owned state

  • Worker ID and assignment scope
  • USER.md, MEMORY.md, and memories/
  • skills/private/ Private Playbooks
  • Sessions, state databases, workspace, and Work History
REPLACED OR ARCHIVED

Release-scoped governed state

  • Role Skills become the new reviewed baseline
  • Candidate Improvements move to the previous-release archive
  • Provenance moves with the release archive
  • Fresh sessions use the new skill tree
Refresh cannot discard unexported governed learning.

If Role Skills differ from baseline or candidates exist, installation of the newer release requires a valid approved export receipt for the current exact state.
