Autopilots on Azure · implementation architecture

A thin control plane. A durable Worker.

This is the current, deployed architecture: Microsoft Teams and Agent 365 at the edge, one scale-to-zero bridge per Worker, Hermes inside an Azure Container Apps Sandbox, durable private state on an attached Data Disk, Service Bus as the wake-up clock, and identity-aware loopback proxies for Foundry and MCP.

Why two regions?

Foundry and ACA Sandboxes run in Sweden Central. Container Apps and ACR run in North Europe because the target managed-environment capacity was unavailable in Sweden Central. Peering and private DNS preserve the intended private MCP path.

03 / RUNTIME

The Sandbox is replaceable. The Worker is not.

The runtime image is immutable and disposable. The profile on the named Data Disk is the durable Worker identity and state boundary.

01 · PROVISION

Create the Data Disk once

The bridge asks the Sandbox Group data plane for the Worker’s named volume. If it does not exist, Azure creates it before the Sandbox.

02 · ATTACH

Mount at /data

The Sandbox mounts the persistent volume. Hermes activates /data/hermes/profiles/<role> and uses its workspace, SQLite state, memories, skills, cron files, and receipts.

03 · REPLACE SAFELY

Labels decide reuse

Worker ID, Role Release, runtime image, and config revision label the Sandbox. A mismatch replaces compute while reattaching the same disk.

04 / TEAMS

Agent 365 outside. Hermes inside.

Teams never connects directly to the Sandbox. The per-Worker bridge terminates Activity Protocol, establishes the privacy boundary, and translates to a runtime-neutral request.

INTERACTIVE TURN
Teams sends an activity

Personal chat, explicit mention, reaction, or supported targeted activity reaches Agent 365.

Agents SDK authenticates ingress

The bridge identifies the Worker, user, conversation type, and selected authorization mode.

Bridge wakes the Sandbox

It creates, resumes, or reuses the labeled Sandbox and waits for health.

Hermes runs the turn

The Gateway owns the session, model calls, tools, memory, skills, and learning transaction.

Bridge returns the activity

Text, reactions, and supported Teams behavior are sent through the same Agent 365 connection.

PROACTIVE SCHEDULED RESULT
Capture the conversation reference

A user-created schedule is durably bound to its existing personal, group, or channel conversation.

Execute after wake

Service Bus starts the bridge; Hermes records the result on the Data Disk.

Continue the conversation

The bridge uses app-ID continuation when persisted inbound claims are empty.

Require an activity ID

The queue message is acknowledged only after Teams returns a concrete activity ID.

Current boundary

Proactive continuation can post into an existing installed conversation. Creating a brand-new chat or contacting a new person belongs to Work IQ Teams/Graph actions, not Activity Protocol continuation.

USER TASK · hermes.cron.fire

Prompt execution with proactive delivery

Revision checks reject stale updates. Pre-fire receipts, Hermes claims, execution leases, bounded delivery receipts, and a required Teams activity ID make redelivery safe.

PLATFORM TASK · system.dream

Same wake path, no agent prompt

A reserved daily Hermes schedule dispatches the existing Dreaming coordinator. Two-phase system receipts provide at-most-once execution and packet-preparation evidence. The former ACA scheduled Job is gone.

Why not an always-running bridge?

The bridge and Sandbox can both reach zero between occurrences. Service Bus scheduled delivery provides a durable future wake without polling, one ARM Job per task, or an in-process timer.

06 / IDENTITY

Execution identity is not business identity.

Managed identity proves where code runs. Agent Identity authorizes autonomous application work. Agent User represents the digital teammate for Worker-owned Microsoft 365 resources.

WORKLOAD CREDENTIAL

Sandbox Group / bridge managed identity

Bootstraps Azure access, Sandbox lifecycle, ACR, Service Bus receive, and the first federated exchange. It is not the business principal exposed to tools.

AUTONOMOUS PRINCIPAL

Agent Identity

Obtains resource-specific application tokens for Foundry, private incidents, public shipments, and Service Bus scheduling. Servers validate audience, tenant, issuer, object/client identity, and app role.

DIGITAL TEAMMATE

Agent User

Uses user_fic to access licensed Worker-owned Microsoft 365 resources, currently demonstrated with Work IQ Mail. It is not the invoking human.

07 / MODEL

Hermes sees OpenAI. Azure sees identity.

Hermes expects an OpenAI-compatible endpoint and key. A small loopback proxy preserves that interface while acquiring Foundry tokens with managed identity.

Hermes agentOPENAI_BASE_URL=http://127.0.0.1:18080/v1
foundry_token_proxy.pyAcquire and cache Azure token; forward OpenAI-compatible request
Foundry OpenAI v1gpt-5-6-terra, managed authorization, no model API key on disk
Why a proxy?

It is a compatibility adapter, not a behavior layer. Hermes still owns prompts, sessions, tools, and inference decisions. The proxy only changes authentication and endpoint transport.

09 / NETWORK

Control plane public. Private data plane private.

The architecture deliberately separates user-facing ingress from private tool reachability. The bridge is not a relay for MCP traffic.

NORTH EUROPE APPLICATION VNET

Bridge and MCP hosting

Public bridge Container Apps environment and an internal private-MCP environment. ACR also resides in North Europe.

SWEDEN CENTRAL SANDBOX VNET

Worker network interfaces

Delegated Sandbox subnet connects through the Sandbox Group VNet connection. Global peering reaches the private MCP environment; linked private DNS resolves its internal FQDN.

Egress inspection remains active

Python HTTP clients use the system trust bundle through SSL_CERT_FILE and REQUESTS_CA_BUNDLE. The Sandbox egress proxy may terminate and re-establish inspected TLS.

11 / FAILURE MODEL

Durability is explicit, not hopeful.

Scale-to-zero and distributed delivery introduce retries and races. The implementation records enough state to distinguish stale work, duplicate work, interrupted work, and delivered work.

QUEUE

PeekLock + renewal

Complete only after durable execution and required delivery. Retry transient failures; dead-letter bounded terminal failures for operator inspection.

HERMES

Claims + revisions

Schedule revision rejects stale cancellation/update races. Claims and receipts prevent redelivery from rerunning an occurrence.

DREAMING

At-most-once occurrence

claiming → running → completing → completed. An expired running lease records interruption and re-arms; it does not duplicate Dreaming side effects.

TEAMS

Activity ID required

No success-shaped acknowledgement. A proactive send must return a concrete activity ID before its queue message completes.

SANDBOX

Replace compute, retain disk

Config/image/identity label drift replaces the Sandbox. Worker state remains on the named Data Disk.

OBSERVABILITY

Sanitized persistent diagnostics

Queue counters, provider schedule, bindings, execution receipts, system receipts, and packet summaries exclude prompts and token material.

Repeatable proof

scripts.user_schedule_smoke proves user-task scale-to-zero and exact output. scripts.servicebus_dream_smoke enqueues a unique operator occurrence, proves Dreaming and packet preparation, and leaves the production daily schedule unchanged.
