A digital Worker that learns without leaking.

Hermes gives each Worker durable memory and native skills. Azure gives it hosting, identity, networking, and lifecycle. GitHub turns selected local lessons into reviewed Role Releases.

Microsoft Agent 365 ACA Sandboxes Hermes-native learning Entra identity GitHub Promotion

We do not call shared versions “generations.” The agreed term is Role Release. A named digital teammate is a Worker, and adopting a newer Role Release is Worker Refresh.

01 / START HERE

The 90-second story

One shared job definition creates multiple private digital teammates. They learn locally, but only approved, generalized evidence can improve the next shared release.

01

Born from a role

A Worker starts from an immutable Role Release containing reviewed instructions and Role Skills.

02

Works privately

Its Personal Memory, Private Playbooks, Work History, identity, and Data Disk belong only to that Worker.

03

Learns locally

Hermes can create Candidate Improvements or patch Role Skills while preserving provenance and privacy boundaries.

04

Improves together

Selected lessons become signed Learning Packets, pass Collective Learning Review, and enter a human-reviewed Promotion PR.

Why not use the human’s Teams token for everything?
Human delegated access is valid only for an explicit user-owned request and consented turn. A group chat cannot make one person’s delegated token ambient authorization for everyone. Autonomous work therefore uses Agent Identity; Worker-owned Microsoft 365 data uses Agent User.
05 / LANGUAGE

The naming convention

Consistent words make the lifecycle understandable to non-technical audiences and precise enough for implementation.

Role Blueprint

The reviewed central definition of a job: instructions, Role Skills, tools, and release metadata.

Role Release

An immutable semantic version of a Role Blueprint, pinned to an exact Git commit.

Worker

A named digital teammate with its own identity, assignment, private state, and local learning.

Personal Memory

Small private facts and preferences injected into every fresh session.

Private Playbook

Rich assignment-specific knowledge loaded only when relevant.

Work History

Private sessions, messages, tool calls, and search indexes used for recall and Dreaming.

Role Skill

A reviewed reusable skill inherited from a Role Release.

Candidate Improvement

A local reusable capability that may—but does not have to—benefit other Workers.

Dreaming

Offline reflection across Work History, outcomes, memory, and skills.

Learning Packet

An approved-boundary export containing allowed artifacts plus provenance.

Promotion

Human-reviewed acceptance of shared learning into a new Role Release.

Worker Refresh

Forward-only adoption of a newer Role Release while preserving private Worker state.

06 / MEMORY

Private state vs. shareable learning

The central idea is not “public memory.” Private information stays private. A separate, generalized artifact may become a Candidate Improvement.

Private Worker plane

Survives Worker Refresh. Never exported raw.

Private Playbooks skills/private/ · progressive disclosure
Work History SQLite sessions and indexes · recall + Dreaming evidence
Assignment workspace Documents, working files, caches, and local operational state
Privacy boundary

Governed learning plane

Generalized, provenance-bound, and reviewable.

records.jsonl is not the Worker’s behavior

The skill tree contains behavior. The journal explains why an eligible skill changed and binds that change to evidence, hashes, Worker identity, and Role Release.

07 / CLASSIFICATION

How Hermes decides what to remember

Every durable observation is classified before storage. The safest useful store wins; uncertain or secret material is not stored as reusable learning.

08 / PROGRESSIVE DISCLOSURE

Skills load only when useful

Hermes indexes skill names and descriptions. Full instructions are loaded only when the task matches—or when a user invokes a skill explicitly with /name.

skills/role/delivery-commitment-control/SKILL.md

Role Skill

Inherited from an immutable Role Release. This example loads when a formal decision or dependency handoff affects delivery.

  • Defines required decision and handoff fields.
  • Defines verification states: pending, accepted, rejected.
  • Defines when and how to escalate.
  • Can be patched locally, but the baseline in Git remains immutable.
What if someone creates a skill directly with hermes --cli?
Direct CLI writes bypass the bridge transaction. On the next bridged turn or Dreaming run, governed drift is quarantined, the committed tree is restored, and safe content is recreated with provenance. Private content becomes a Private Playbook; rejected content stays quarantined.
09 / REFLECTION

Dreaming expands the evidence horizon

Dreaming is not “always create a skill.” It is a scheduled or manual reflection that may consolidate private memory, improve a playbook, create transferable learning, suppress a duplicate, or intentionally store nothing.

1. Wake or reuse the Worker

The bridge manages the ACA Sandbox lifecycle and waits for real Hermes API readiness.

2. Open an isolated Dream session

Dreaming uses a fresh session with a bounded focus and maximum artifact count.

3. Compare Work History and local state

Hermes can reason across outcomes, corrections, tool traces, Personal Memory, Private Playbooks, Role Skills, and candidates.

4. Classify every observation

Private facts remain private. Reusable behavior becomes a Role Skill patch or Candidate Improvement with provenance.

5. Optionally prepare a packet

Scheduled learning may prepare an approval-required digest when transferable records exist—but cannot approve or export it.

CLASSROOM PATH

Bridge-owned timer

Simple and inspectable. One bridge replica remains active and runs the same serialized Worker learning transaction.

PRODUCTION PATH

ACA scheduled Job

A managed-identity Job calls a protected bridge operation. No Worker key or bridge API key is stored in the Job.

10 / COLLECTIVE LEARNING

From local lessons to shared behavior

Local learning is fast and allowed to be imperfect. Collective Learning Review is slower, compares multiple Workers, minimizes evidence, resolves conflicts, and proposes the smallest coherent shared change.

Evidence can be repeated, complementary, conflicting, or unrelated. The retained decision must say which—and never claim more support than the packets provide.

11 / GITHUB GOVERNANCE

The PR is reviewed twice

Deterministic code validates signatures, hashes, paths, schemas, and versions. Independent GitHub Agentic Workflows review semantic quality using read-only reasoning and permission-separated safe outputs.

Triage

Promotion Triage

Labels the PR and summarizes release scope, Workers, records, conflicts, and required gates.

Privacy

Privacy Guardian

Checks for people, projects, identifiers, raw messages, private instructions, and reconstructable source detail.

Mission

Role Alignment

Compares the proposal with the Role Blueprint mission and rejects unrelated or overpowered capabilities.

Evidence

Evidence Auditor

Verifies that claims remain within retained Worker evidence and account for conflicts and rejections.

Quality

Skill Quality

Checks packaging, discovery trigger, standalone execution, fields, states, gates, overlap, and consistency.

Agent job

Read-only repository and PR context. It can reason, but cannot directly modify the repository or merge the Promotion.

Safe output job

Applies only allowlisted labels, comments, and reviews after threat detection and structural validation.

Why create a draft PR instead of a ready PR?
The merger/judge is proposing generalized behavior, not approving it. Draft status communicates that semantic gates and human expert review must finish before Promotion. The automation cannot mark the PR ready, merge it, or perform Worker Refresh.
12 / FORWARD-ONLY ADOPTION

Worker Refresh changes the role, not the person

After Promotion merges, each Worker adopts the strictly newer Role Release. Reviewed shared behavior is replaced; private state stays with the Worker.

Preserved

✓ Worker ID and assignment
✓ Agent Identity and Agent User
✓ Personal Memory
✓ Private Playbooks
✓ Work History and Data Disk

Replaced or archived

→ Role Skills replaced by the new reviewed baseline
→ Previous Candidate Improvements archived
→ Previous-release provenance archived
→ Fresh sessions use the new skill index
→ New Sandbox may reuse the same Data Disk
Preflight Newer semantic version + approved export if governed state changed
Verify Digest + signature + Worker + release + current state hash
Refresh Transactional profile update with recoverable backup
Validate Authoritative health + fresh-session behavior + private markers
Failure is safe

If preflight, signature, state, or copy validation fails, the current Worker remains available. Refresh never silently discards local learning.

Suggested teaching sequence
  1. Start with the 90-second story and architecture.
  2. Show the Worker in Teams, then explain the three identities.
  3. Teach one Personal Memory fact and one Private Playbook.
  4. Create distinct Candidate Improvements in two Workers.
  5. Run Dreaming and inspect canonical files plus provenance.
  6. Prepare, approve, and export both Learning Packets.
  7. Show the merger/judge decision and draft Promotion PR.
  8. Walk through the five Agentic Promotion gates.
  9. Refresh both Workers and prove private state survived.
  10. Reset the disposable cohort for the next audience.
14 / DETAILS

The important invariants

These are the rules that make the demonstration more than an AI story: they are enforced by code, cryptography, namespaces, Terraform state, and explicit human gates.

Single writer

Learning operations serialize per Worker because Hermes uses SQLite and transactional skill state.

Immutable baseline

Every Role Release is a semantic version pinned to a full Git commit with baseline Role Skill hashes.

Fail-closed export

Only allowlisted Role Skill diffs and Candidate Improvements with matching current-hash provenance can enter a packet.

Independent approval

The bridge holds the Ed25519 private key. The Worker cannot forge human packet approval.

Deterministic review context

Every semantic PR reviewer receives the exact same prefetched PR-head metadata, diff, history, and Role Blueprint snapshot.

No automatic Promotion

Dreaming, packet preparation, merger/judge, and Agentic reviewers cannot merge or refresh Workers.
