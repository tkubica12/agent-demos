Digital Workers learn. The role evolves.

Hermes gives each Worker durable memory and native skills. Azure provides serverless hosting, networking, and model intelligence. Microsoft 365 provides the user experience, enterprise identity, and security. GitHub turns selected local lessons into reviewed Role Releases.

01 / START HERE

The Continuous Improvement Loop

One shared Role Blueprint creates multiple private digital teammates. They learn locally, but only approved, generalized evidence can improve the next shared release.

01

Born from a role

A Worker starts from an immutable Role Release containing reviewed instructions and Role Skills.

02

Works privately

Its Personal Memory, Private Playbooks, Work History, identity, and Data Disk belong only to that Worker.

03

Learns locally

Hermes can create Candidate Improvements or patch Role Skills while preserving provenance and privacy boundaries.

04

Improves together

Selected lessons become Learning Packets, pass Collective Learning Review, and enter a human-reviewed Promotion PR. Merge creates the next Role Release, then Workers Refresh and the loop begins again.

Human On-Behalf-Of: private, explicit, and still deferred

OBO means the Worker receives and exchanges a human’s delegated token to access that human’s emails, calendar, or files. A private Teams message gives the right conversation boundary, but is not enough by itself: the specific user must explicitly request user-owned data, authenticate or consent, and provide a delegated token for that turn.

Our bridge already carries sender and privacy-boundary metadata, but the delegated-token acquisition path is not implemented. If added, results must stay targeted to that user. Group chats and channels must never inherit one participant’s OBO token or receive their private data.

06 / LOCAL LEARNING

Two ways a Worker learns locally

Hermes can learn immediately during work or reflect later across several sessions. Both paths use the same privacy classification and provenance rules.

NATIVE HERMES · FOREGROUND

Immediate learning

An explicit correction or /learn request can update Personal Memory, create or patch a Private Playbook, patch a Role Skill, or create a Candidate Improvement during the current task.

NATIVE HERMES · REFLECTION

Dream-time learning

Dreaming compares Work History, outcomes, corrections, memory, playbooks, and skills. It can update Personal Memory, create or patch a Private Playbook, patch a Role Skill, create a Candidate Improvement, suppress a duplicate, or intentionally store nothing.

Custom governance enhancements around Hermes

Reserved private/role/candidate namespaces, serialized transactions, snapshots and rollback, provenance bound to file hashes, quarantine, privacy-scanned Learning Packets, owner approval, and forward-only Role Releases.

A Candidate Improvement is the actual SKILL.md file—not a row in the journal. The matching learning/records.jsonl entry is its provenance: why it changed, summarized evidence, confidence, hashes, Worker, and Role Release. Private Playbook and Personal Memory changes do not create transferable provenance rows.

07 / SHARED EVOLUTION

From local learning to a new Role Release

Shared evolution has three explicit phases and two reviewable artifacts. Nothing moves directly from a Worker into the shared Role Blueprint.

PHASE 1 · WORKER-LOCAL

Select and approve

Eligible skill change + provenance → allowlist and privacy rules → exact digest reviewed by the Worker owner/operator → bridge attestation.

Artifact: Learning Packet
PHASE 2 · CENTRAL REASONING

Merge and judge

Today this physically runs in the operator process executing scripts/collective_review.py—typically a workstation or CI runner, not either Worker Sandbox. The script calls Azure Foundry, compares approved packets with the current Role Blueprint, and distinguishes repeated, complementary, conflicting, or unrelated evidence.

Artifact: Draft Promotion PR
PHASE 3 · GITHUB GOVERNANCE

Review and promote

The PR stays draft while deterministic checks and read-only semantic agents run. A human expert is the final approval gate before merge creates the next immutable Role Release.

Artifact: Role Release N+1

The Promotion PR approval process

Triage

Triage Agent

Labels the PR and summarizes release scope, Workers, records, conflicts, and required gates.

Privacy

Privacy Agent

Checks for people, projects, identifiers, raw messages, private instructions, and reconstructable source detail.

Mission

Role Alignment Agent

Compares the proposal with the Role Blueprint mission and rejects unrelated or overpowered capabilities.

Evidence

Evidence Agent

Verifies that claims remain within retained Worker evidence and account for conflicts and rejections.

Quality

Skill Quality Agent

Checks packaging, discovery trigger, standalone execution, fields, states, gates, overlap, and consistency.

Human

Human Expert Approval

Reviews the complete proposal and agent findings, marks the PR ready, and decides whether Promotion should merge.

08 / FORWARD-ONLY ADOPTION

Worker Refresh

After Promotion merges, each Worker adopts the strictly newer Role Release. Reviewed shared behavior is replaced; the Worker’s identity and private continuity stay intact.

Preserved

✓ Worker ID and assignment
✓ Agent Identity and Agent User
✓ Personal Memory
✓ Private Playbooks
✓ Work History and Data Disk

Replaced or archived

→ Role Skills replaced by the new reviewed baseline
→ Previous Candidate Improvements archived
→ Previous-release provenance archived
→ Fresh sessions use the new skill index
→ New Sandbox may reuse the same Data Disk
The implementation is intentionally robust

Before replacing anything, Refresh requires a strictly newer semantic release and—when local governed learning exists—an approved packet bound to the exact Worker, release, signature, and current state hash. The profile update uses a recoverable backup. If preflight or copy validation fails, the current Worker remains available.
